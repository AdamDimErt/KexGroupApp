💰 Cost Allocation Engine

Алгоритм распределения затрат ГО
Коэффициент = Выручка_точки / Σ Выручка_все_точки Распределённая_затрата = Затрата_ГО × Коэффициент Финрезультат_точки = Выручка − Прямые_расходы(iiko) − Распределённые_затраты(1C)
Шаги выполнения
1
Получить выручку каждого ресторана
Из FinancialSnapshot за выбранный период (данные iiko OLAP)
2
Посчитать общую выручку
Сумма по всем 13 точкам = знаменатель для коэффициента
3
Рассчитать коэффициент каждой точки
coefficient = restaurant.revenue / total_revenue (0.0 — 1.0)
4
Распределить затраты ГО из 1C
Для каждой статьи расходов ГО: allocated = hq_expense × coefficient
5
Сохранить в Expense (ONE_C_ALLOCATED)
Пересчитывать при каждой смене периода или новых данных выручки

⚠️ Ограничения и важные нюансы

🚦
Rate Limiting
Большинство reporting эндпоинтов ограничены окном запроса 15-30 дней. Для исторических данных нужна стратегия windowing: запрашивать по 15-дневным отрезкам. При превышении — exponential backoff.
📄
Пагинация
Большие результаты (чеки, расходы за долгий период) требуют пагинации через offset + limit. Реализовать в aggregator-worker итеративный сбор всех страниц.
⏰
Время жизни токена — 1 час
JWT токены истекают через 1 час. Необходимо автоматическое обновление каждые 45 минут. Хранить токен в Redis с TTL 55 минут. SDK-библиотеки (TypeScript/Python/Go) делают это автоматически.
⚡
OLAP — долгая генерация
Сложные OLAP отчёты могут занимать несколько секунд. Реализовать асинхронный запрос с polling статуса. Кэшировать результаты в Redis с TTL 15 минут.
🔄
Eventual Consistency
Данные поступают с задержкой. Polling 15 мин (выручка), 30 мин (ДДС), 1 час (касса, цех) вносит latency. Дашборд должен показывать время последнего обновления.
🌍
Географическая задержка
API базируется на серверах в России (api-ru.iiko.services). Latency из Казахстана ~200-500ms. Учитывать в timeout настройках HTTP клиента (рекомендуется 30 секунд).
