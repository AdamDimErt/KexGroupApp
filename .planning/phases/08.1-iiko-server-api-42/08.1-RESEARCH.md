## Summary

Phase 08.1 connects real DDS expense articles from iiko Server API to the finance DB. The current `syncExpenses()` method calls `/reports/productExpense`, which returns ingredient-level food costs (product consumed per restaurant) — NOT the 42 DDS cash movement articles (Аренда, ЗП, Комиссия банка, etc.). These are fundamentally different data types. This phase introduces a new `syncDdsTransactions()` method that (a) fetches the DDS account list from `/v2/entities/list?rootType=Account`, upserts `DdsArticle` records, (b) fetches cash shift payIn/payOut transactions from `/v2/cashshifts/list`, and (c) upserts `Expense` records with syncId deduplication.

The existing `Expense` model schema requires NO changes — it already has all fields: `articleId`, `restaurantId`, `date`, `amount`, `comment`, `source`, `syncId`. All infrastructure (circuit breaker, retry, XMLParser, makeRequest/makePostJsonRequest, logSync) is reusable from the existing service.

**Primary recommendation:** Add `syncDdsTransactions()` as a new method in `IikoSyncService`. Do NOT remove or modify `syncExpenses()` — food product expenses and DDS cash article expenses are separate data tracks. The DDS transaction endpoint schema must be validated against live server response before hardcoding field names.

### Pattern 1: Two-step DDS sync (accounts → transactions)

**What:** First sync the account/article catalog, then sync transactions that reference those accounts.
**When to use:** Any time the transaction records reference FK to DdsArticle — must ensure articles exist before inserting Expense rows.

```typescript
// Step 1: Sync DDS article catalog
async syncDdsArticles(): Promise<void> {
  // GET /v2/entities/list?rootType=Account (POST JSON, key in query)
  const response = await this.makePostJsonRequest('/v2/entities/list', {
    rootType: 'Account',
  });
  // response.items: [{ id, name, code, parentId, ... }]
  // Map each account to a DdsArticleGroup by name matching
  for (const account of response.items) {
    const groupCode = this.resolveGroupCode(account.name);
    const group = await this.prisma.ddsArticleGroup.findFirst({
      where: { code: groupCode, tenantId: this.tenantId },
    });
    await this.prisma.ddsArticle.upsert({
      where: { code: account.id }, // code = iiko account UUID
      update: { name: account.name, isActive: !account.isDeleted },
      create: {
        groupId: group.id,
        name: account.name,
        code: account.id,
        source: 'IIKO',
        allocationType: 'DIRECT',
        isActive: true,
      },
    });
  }
}
```

### Pattern 2: resolveGroupCode — name-to-group mapping

**What:** Deterministic function mapping iiko account name (Russian) to one of 12 DB group codes.
**When to use:** During syncDdsArticles to assign groupId.

```typescript
private resolveGroupCode(accountName: string): string {
  const name = accountName.toLowerCase();
  if (name.includes('аренд')) return 'RENT';
  if (name.includes('зарплат') || name.includes('зп') || name.includes('оплата труда')) return 'SALARY';
  if (name.includes('коммисси') || name.includes('банк')) return 'BANK_FEE';
  if (name.includes('коммунал') || name.includes('электр') || name.includes('вода')) return 'UTILITIES';
  if (name.includes('маркетинг') || name.includes('реклам')) return 'MARKETING';
  if (name.includes('it') || name.includes('програм') || name.includes('интернет')) return 'IT';
  if (name.includes('транспорт') || name.includes('доставк')) return 'TRANSPORT';
  if (name.includes('оборуд') || name.includes('ремонт')) return 'EQUIPMENT';
  if (name.includes('налог') || name.includes('штраф')) return 'TAXES';
  if (name.includes('кухн') || name.includes('производств')) return 'KITCHEN';
  if (name.includes('продукт') || name.includes('еда') || name.includes('food')) return 'FOOD';
  return 'OTHER';
}
```

### Pattern 3: syncDdsTransactions — cash shift payIn/payOut fetch

**What:** Fetch daily DDS cash transactions per restaurant from `/v2/cashshifts/list`.
**When to use:** Every 30 min on cron schedule.

CRITICAL: The exact field names in `/v2/cashshifts/list` response are LOW confidence — must be validated against live server. The pattern below uses hypothetical field names based on iiko Server API conventions:

```typescript
async syncDdsTransactions(dateFrom: string, dateTo: string): Promise<void> {
  for (const restaurant of this.restaurants) {
    // LOW CONFIDENCE: exact request body and response schema TBD
    const response = await this.makePostJsonRequest('/v2/cashshifts/list', {
      organizationId: restaurant.iikoId,
      dateFrom,
      dateTo,
    });
    // response.cashShifts: [{ id, restaurantId, date, payIns: [...], payOuts: [...] }]
    for (const shift of response.cashShifts ?? []) {
      for (const movement of [...(shift.payIns ?? []), ...(shift.payOuts ?? [])]) {
        // movement: { id, accountId, amount, comment, type: 'PAY_IN'|'PAY_OUT' }
        const syncId = `dds:${shift.id}:${movement.id}`;
        const article = await this.prisma.ddsArticle.findUnique({
          where: { code: movement.accountId },
        });
        if (!article) continue; // unknown account — skip or log
        await this.prisma.expense.upsert({
          where: { syncId },
          update: { amount: movement.amount, comment: movement.comment },
          create: {
            articleId: article.id,
            restaurantId: restaurant.id,
            date: new Date(shift.date),
            amount: movement.amount,
            comment: movement.comment ?? null,
            source: 'IIKO',
            syncId,
          },
        });
      }
    }
  }
}
```

### Pattern 4: Reuse existing makePostJsonRequest

**What:** All v2 endpoints use POST JSON with `?key={token}` appended by makePostJsonRequest.
**When to use:** For both `/v2/entities/list` and `/v2/cashshifts/list`.

The method signature (already in IikoSyncService):
```typescript
private async makePostJsonRequest(
  endpoint: string,
  body: Record<string, unknown>,
): Promise<unknown>
```
Returns parsed JSON. No changes needed.

### Pattern 5: syncId deduplication

**What:** Every Expense row has a unique `syncId` string. On upsert, if syncId exists → update amount/comment; if not → insert. Prevents duplicate rows across repeated 30-min syncs.
**Formula:** `dds:{shiftId}:{movementId}` — ensures global uniqueness across restaurants, days, and articles.

### Anti-Patterns to Avoid

- **Reusing `/reports/productExpense` for DDS:** This endpoint returns food ingredient costs per day per dish, not financial account movements. Keep them separate.
- **Removing `syncExpenses()`:** The existing food cost sync is still needed (KitchenShipment/FOOD article expenses). Do not remove or repurpose it.
- **Hardcoding account UUIDs:** iiko account UUIDs differ per environment. Always fetch the list dynamically.
- **Missing tenantId on DdsArticleGroup lookup:** `DdsArticleGroup` has `@@unique([tenantId, code])` — always filter by tenantId when resolving groups.

## Common Pitfalls

### Pitfall 1: Confusing productExpense with DDS transactions
**What goes wrong:** Dev assumes current `syncExpenses()` already syncs DDS articles. It does not — it syncs food/ingredient costs.
**Why it happens:** Both return "expenses" from iiko, but at completely different abstraction levels.
**How to avoid:** Keep both methods. Name new method `syncDdsTransactions()` to make the distinction explicit.
**Warning signs:** DdsArticle records appear with `name` containing dish names (e.g., "Бургер классический") rather than account names (e.g., "Аренда помещения").

### Pitfall 2: Unknown `/v2/cashshifts/list` response schema
**What goes wrong:** Field names assumed from training data don't match actual iiko Server response.
**Why it happens:** iiko Server API is not publicly documented; v2 JSON endpoints vary by server version.
**How to avoid:** In Wave 0 or first task, add a raw logging call to print the actual response before building upsert logic. Validate field names against live server.
**Warning signs:** `movement.accountId` is undefined; `shift.payIns` is null; `response.cashShifts` is empty despite active restaurants.

### Pitfall 3: Missing restaurantId on Expense for distributed articles
**What goes wrong:** Some DDS accounts are HQ-level (not restaurant-specific). Setting `restaurantId = null` may break Cost Allocation Engine expectations.
**Why it happens:** iiko Server returns organization-level accounts alongside restaurant-level accounts.
**How to avoid:** Flag HQ-level accounts as `allocationType: DISTRIBUTED` on DdsArticle. Cost Allocation Engine already handles null restaurantId on Expense.
**Warning signs:** Multiple restaurants show identical expense amounts from same account on same day.

### Pitfall 4: Stale article cache when running syncDdsTransactions before syncDdsArticles
**What goes wrong:** `prisma.ddsArticle.findUnique({ where: { code: movement.accountId } })` returns null → expense skipped silently.
**Why it happens:** On first run, DdsArticle table may be empty.
**How to avoid:** Always call `syncDdsArticles()` before `syncDdsTransactions()` in the cron handler. Or use `findUnique` and emit a warning to Sentry if article not found.

### Pitfall 5: tenantId hardcoded vs. env-variable
**What goes wrong:** DdsArticleGroup lookup fails if tenantId is wrong.
**Why it happens:** seed.ts uses a specific tenantId; the service may use a different one.
**How to avoid:** Read tenantId from `process.env.TENANT_ID` or look it up dynamically from the first Restaurant record.

private async makePostJsonRequest(
  endpoint: string,
  body: Record<string, unknown>,
): Promise<unknown> {
  const token = await this.authService.getAccessToken();
  const url = `${this.baseUrl}${endpoint}?key=${token}`;
  const response = await firstValueFrom(
    this.httpService.post(url, body, { timeout: 30000 }),
  );
  return response.data;
}

## Open Questions

1. **Exact field names in `/v2/cashshifts/list` response**
   - What we know: Endpoint exists, returns shift-level data with payIn/payOut per pre-verified context
   - What's unclear: JSON field names (`payIns`? `incomings`? `accountId`? `account`?), pagination support, date filter parameter names
   - Recommendation: In Wave 0 task, add a debug call that logs raw response JSON from live server. Build upsert logic only after field names confirmed.

2. **Does `/v2/entities/list?rootType=Account` require POST JSON or GET with query param?**
   - What we know: Pre-verified context says `GET /v2/entities/list?rootType=Account` — but existing v2 methods use POST JSON
   - What's unclear: Whether rootType is a query param on GET or a body field on POST
   - Recommendation: Try GET first (`makeRequest('/v2/entities/list?rootType=Account')`). If 405, switch to POST body `{ rootType: 'Account' }`.

3. **Are HQ-level DDS accounts included in restaurant shift data?**
   - What we know: Some of the 42 articles may be HQ overhead (no restaurantId)
   - What's unclear: Whether `/v2/cashshifts/list` returns them per restaurant or only per shift
   - Recommendation: Mark accounts with no parentRestaurantId as `allocationType: DISTRIBUTED`. Cost Allocation Engine handles them.
