# Мобильный Дашборд Руководителя: Спецификация

## Executive Summary
Мобильное приложение (React Native) и бэкенд (NestJS + PostgreSQL), агрегирующие финансовые и операционные данные из iiko и 1С. Цель — дать директорам ресторанного холдинга оперативную картину по всем заведениям в реальном времени в удобном мобильном формате, избавив от необходимости вручную выгружать отчеты из десктопных систем.

## Problem Statement
В данный момент директора и топ-менеджмент не имеют удобного инструмента для быстрого контроля ключевых метрик. Данные разрознены: кассовые/операционные показатели живут в iiko, банковские/бухгалтерские остатки — в 1С. Сбор сводной информации занимает время, что замедляет принятие управленческих решений.

## Success Criteria
- Ежедневное использование приложения директорами (высокий DAU).
- Сокращение времени на получение сводной отчетности с часов/дней до секунд.
- Отсутствие жалоб на рассинхрон данных между дашбордом и первоисточниками (iiko/1С).

## User Personas
- **Директора / Владельцы (Топ-менеджмент)**: Не технические специалисты, ценят свое время, решают бизнес-задачи. Им нужна максимально наглядная картина: "где хорошо, где плохо".
- *(Потенциально)* **Управляющие точками**: Могут получить ограниченный доступ только к своим ресторанам в будущем.

## User Journey
1. Директор открывает приложение.
2. Проходит авторизацию по номеру телефона (OTP SMS/Звонок).
3. Видит **список всех ресторанов с цветовыми индикаторами (зеленый/красный)**, что позволяет за 1-3 секунды оценить общую ситуацию.
4. Выбирает проблемный (красный) ресторан, чтобы углубиться в детали (выручка, фудкост, остатки на счетах и т.д. за выбранный период).

## Functional Requirements
### Must Have (P0)
- Авторизация по номеру телефона.
- Агрегация данных продаж и метрик из iiko (Pull-модель по расписанию).
- Агрегация остатков/движений по счетам из 1С (Pull-модель HTTP сервисов/OData).
- Главный экран со списком заведений и индикацией отклонений (План/Факт или Вчера/Сегодня).
- Развернутый экран по конкретному заведению с графиками/виджетами.

### Should Have (P1)
- Настройка порогов для "красных/зеленых" индикаторов (через админку или конфиг).
- Уведомления (Push) о критических отклонениях или кассовых разрывах.
- Ролевая модель (CFO видит всё, Управляющий — только свою точку).

## Technical Architecture

### Монорепозиторий
- **Инструмент**: Turborepo (оптимизация сборки, шаринг типов и конфигураций между сервисами).

### Frontend (Мобильное приложение)
- **Технологии**: React Native (или Expo для быстрого старта).
- **Платформы**: iOS и Android.

### Backend (Микросервисы NestJS)
1. **`api-gateway`**: Единая точка входа для мобильного приложения. Маршрутизирует REST/GraphQL запросы, агрегирует ответы.
2. **`auth-service`**: Управляет сессиями, пользователями, ролями. Интеграция со шлюзами SMS/Звонков для OTP.
3. **`finance-service`**: Бизнес-логика финансов. Отдает готовые агрегаты, кэширует тяжелые расчеты.
4. **`aggregator-worker`**: Фоновый процесс (CRON/BullMQ). Регулярно ходит в iiko (iikoCloud API) и 1С (HTTP-сервисы), нормализует данные и складывает в БД.

### Data Model (База Данных)
- **СУБД**: PostgreSQL.
- **Подход**: Единый кластер БД, но **"Schema-per-service" (Схема на сервис)**. У `auth_service` своя схема, у `finance_service` — своя. Это обеспечит достаточную изоляцию для микросервисов и позволит безопасно генерировать/развивать код независимо.
- **Инструмент доступа**: Prisma ORM или TypeORM.

### Интеграции
- **iiko**: REST API (Pull-модель).
- **1С**: HTTP-сервисы (OData/REST, Pull-модель).
- **СМС-шлюз**: (например, SMS.RU, smsc.ru и т.д.) для авторизации.

### Deployment & Operations
- **Среда**: On-Premise (на локальных серверах компании) либо закрытый контур провайдера с VPN. Требуется настройка защищенного доступа извне для мобильных клиентов.
- **Инструментарий**: Docker Compose для старта, потенциально Kubernetes в будущем.

## Pre-Mortem (Анализ Рисков)

### [TIGER] Риск рассинхрона данных (Data Integrity)
- **Сценарий**: В iiko чек отменили задним числом, а дашборд все еще показывает старую выручку. Директор видит ложные данные.
- **Митигация**: Реализовать надежный механизм сверки (reconciliation). Воркер должен не только догружать новые данные инкрементально, но и раз в день/ночь делать полную сверку за предыдущие N дней.

### [TIGER] Сетевая связность On-Premise (Connectivity)
- **Сценарий**: Бэкенд развернут локально, 1С локально. Мобильное приложение не может достучаться до API из-за настроек корпоративного Firewall (блокировки портов, отсутствие белого IP).
- **Митигация**: На раннем этапе (Dev/Staging) привлечь системного администратора компании для выдачи доступов, настройки обратного прокси (Nginx/Traefik) и SSL-сертификатов на стороне компании.

### [ELEPHANT] Нагрузка на системы-источники
- **Сценарий**: `aggregator-worker` слишком часто и тяжелыми запросами опрашивает 1С, из-за чего 1С начинает тормозить у бухгалтеров.
- **Митигация**: Вынести выгрузку в 1С в регламентные задания на стороне самой 1С (пусть она собирает витрину данных в отдельный регистр), а `aggregator-worker` будет просто забирать готовые "плоские" данные, а не строить сложные отчеты на лету.

## Out of Scope (Что НЕ делаем в первой версии)
- Полноценная замена интерфейсов iiko/1С для ввода данных (приложение только для чтения).
- Интеграция с системами бронирования столов.
- Встроенные мессенджеры/чаты.

3. Нужен ли веб-интерфейс (Админка) для заведения пользователей, или пользователи будут приезжать из Active Directory / 1C?
